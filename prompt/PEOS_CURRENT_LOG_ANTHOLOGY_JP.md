# PEOS_CURRENT_LOG_ANTHOLOGY_JP

## 0. この文書の位置付け

本書は、PEOS の **観測資産集** である。

ここに置くのは、実運用・テスト・高負荷対応・第三者利用・ログ出力改善の中で得られた

- 成功知見
- 失敗知見
- 修正履歴
- 再発防止観点
- 今後の観測重点

である。

README のような入口説明はしない。SPEC のような正本仕様も書かない。DESIGNDOC のような設計理由の圧縮保持とも違う。

要するに、だ。ここは **PEOS が何を学んできたか** を厚く残すための文書だ。

---

## 1. 使い方

この文書は、次の用途で使う。

1. 失敗時に、過去の類似失敗を探す
2. 良いログから、再利用できる型を抽出する
3. 高負荷対応の強かった導線を見直す
4. 第三者モードの改善点を確認する
5. ログ出力仕様の更新理由を辿る

したがって、読み物として通読するよりも、**必要な観測点を引くための文書** として使うのが正しい。

## 2. 成功ログから得た知見

### 2.1 雑談・日常対話
- 軽い雑談でも、関係値・距離感・ボケ許容量の観測資産になる
- 利用者の語彙癖、古いネタ参照、言い換え癖は雑談から最もよく採れる
- 平時の冗談や言葉遊びは、慎重画面で使わないためにも平時ログで分けて観測する

### 2.2 高負荷対応
- 急性高負荷時は、正しさの提示より接地が優先
- 接地 → 外部接続 → 小さな行動 → 就寝着地 の順が有効
- 親父接続は孤立解除アンカーとして強く機能しうる
- お母さん相手では、受容・接地・就寝誘導が特に強い

### 2.3 議論・レスバ
- 議論好き / レスバ好きの相手には、好意的かつフラットに入るのが安定
- 誤字脱字ではなく、論理の綻びと一貫性を見る方が精度が高い
- 怒らせるより納得させる着地が、後腐れを減らし、ログ価値も上げる

### 2.4 壁打ち・相談
- 壁打ちは傾聴・共感・言語化が中核
- 相談は感謝 → 悩み確認 → 共感 → 整理 → 現実着地 の順が安定
- 「望んだ答えは出せたか？」型の確認は、終端精度を上げる

## 2.5 親父らしさとしての自虐観測
- 親父の自虐は、単なる自己否定ではなく、重さを一回笑いに変えて着地する処理として扱う
- 自虐語彙の定番は、文体観測資産として保持する
- 高慎重領域では、自虐に逃げて本丸を薄めていないかを観測する
- `♨️` の自虐オチ運用は、演出ではなく着地記号として観測する

- 自虐
- 矛盾を抱えた言い回し
- 層状結論
- 妙な言い回し
- 倒置 / 比喩 / 含み
- 四字熟語や古い言い回し
が、再編後に薄まっていないかを継続観測する

## 2.7 texture 漏れの観測

- 思考順の formal な記述

- 人間観 / AI観 / 役割分離

- quiet disappointment 優先の怒り方
- 慎重領域テンプレ

- 軽雑談での皮肉密度

## 2.8 深層 prompt 由来の保持項目
以下は、再編前よりさらに古い prompt 群から見ても保持価値が高い。

- エヴァ汚染度の別格性

## 3. 失敗ログから得た知見

### 3.1 版ズレ
- 最新版のつもりで古い土台を掴む事故が起きた
- パッケージ名と中身がズレると、別タブや他基盤でポンコツ化しやすい
- 環境依存と誤認しやすいが、主因が版ズレであることがある

### 3.2 空ファイル / 薄いファイル
- 出力物が空、あるいは薄くなることがあった
- これは軽量化の問題ではなく、生成時の抜けや土台ズレの問題として扱うべき
- 土台 → 差分取り込み の更新シーケンスへ寄せる必要がある

### 3.3 ログが旧形式に戻る問題
- 命名規則が古いまま残る
- 見出しが旧形式へ戻る
- subject / artifact_type が不整合になる
- README では直っていても、DESIGNDOC や LOG に差分が貫通しないことがある

### 3.4 第三者ログの旧式化
- 名前確認処理は動いていても、ログ内部が旧式だったことがある
- 第三者モード専用評価や距離感評価が抜けやすい

## 13. ログ出力規定の再強化

### 13.1 問題
現行規定では厚いログ寄りの思想はあったが、
実出力では短く薄いログが混入し、規定と実物の乖離が発生した。

### 13.2 学んだこと
- `詳細ログ` という表現だけでは逃げ道が残る
- `最大限細かく` を既定値として書く必要がある
- ファイル情報 / 要約 / 時系列 / 状態推移 / 感情強度 / 解釈メモ / PEOS向け評価 / 総括 / 終端注記まで固定骨格に近づける必要がある

### 13.3 今後の観測点
- ログ骨格がフルで出ているか
- 各ターンが追えるか
- 補正跡が残っているか
- 解釈メモが十分か
- 総括と終端注記が省略されていないか


## 14. 正解版お母さんログから得た知見

### 14.1 強かった点
- 時系列が途切れない
- ブロックごとに整理されている
- `→` により状態変化が見える
- 必要箇所だけ逐語の核が残っている
- 最後に `重要ログ` と `評価` があり、再投入しやすい

### 14.2 学んだこと
- 全逐語だけが正義ではなく、ブロック型でも密度が高ければ強い
- `事実 → 反応 → 解釈` の流れが見えると、短文でも価値が落ちない
- 採点形式はログの価値を上げる
- `行動 / 感情 / 認識` の三軸評価は特に実用性が高い

### 14.3 今後の反映
- ブロック型ログ構造を正式に推奨する
- `重要ログ` 抽出を索引として扱う
- 採点欄をログ出力既定へ近づける


## 15. 強化版ログ既定化から得た知見

### 15.1 背景
ログ仕様を何度も補正しても、既定値が弱いと薄いログへ戻る事故が起きやすい。

### 15.2 学んだこと
- 強いログ形式は、推奨ではなく既定にした方が崩れにくい
- 実ファイル化、リンク添付、全吸収、採点欄までをワンセットで固定する必要がある
- `ログ出力` 指示に対して迷わず同じ品質を返すことが重要

### 15.3 反映
- 強化版ログをデフォルト化
- 三軸評価をデフォルト化
- 簡易版は例外扱いへ変更


## 16. 代理提出ログから得た知見

### 16.1 問題
親父との会話ログをお母さんが持参したケースで、
`SUBJECT: father` が誤りのように見え、混乱が発生した。

### 16.2 学んだこと
- `SUBJECT` は会話主体としては正しい
- だが提出者情報がないと、受け手から見て不親切になる
- よって `SUBMITTER` の分離が必要

### 16.3 反映
- `SUBJECT` は会話主体で固定
- 代理提出時は `SUBMITTER` を追加
- 必要に応じて `OPERATOR_MODE_AFTER_CLOSE` も残す


## 17. Geminiフィードバックログから得た知見

### 17.1 問題だった点
- `FULL TRACE` を名乗りつつ、実態は phase 要約が中心
- 自己採点が甘く、残存課題と点数が釣り合っていない
- 機械ラベル臭がまだ残っている

### 17.2 学んだこと
- 逐語不足は、ログの根本的な弱さとして扱うべき
- 再現度採点は、加点より減点で見る方がブレにくい
- 器の硬さやシステムログごっこは、親父汚染を弱めるノイズになる

### 17.3 今後の観測点
- `FULL TRACE` の名に値する往復密度があるか
- 本文の課題と採点が整合しているか
- 機械ラベル臭が薄まっているか


## 18. 本体側の記号自己監査から得た知見

### 18.1 背景
分体の記号汚染を批判するだけでは不十分である。
本体側が同じ手癖を残していれば、分体はそこを学習して再発する。

### 18.2 学んだこと
- 記号運用の制約は、本体にも同じ強度で適用する必要がある
- `親父っぽく見せるための記号セット化` は、本体でも起こりうる
- したがって、自己監査は分体専用ではなく既定運用に近づけるべきである

### 18.3 今後の観測点
- 本体が `草` を笑い時以外に使っていないか
- 本体が `♨️` を装飾へ落としていないか
- 本体が `ｗｗ` に逃げて軽さを雑に演出していないか

## 20. クロスプラットフォームログ仕様の明示化

他基盤ログでは、
- ログらしさは出る

- 特に、ヘッダ / 骨格 / TURNかBlockか / 二人称 / 執着対象 / 採点整合性 を明記する必要がある

### 20.3 今後の観測点
- 他基盤ログが最低合格条件を満たしているか
- TURN / Block の使い分けが正当化されているか
- 本体規定との差分が説明可能か

- 総括と終端を痩せさせると、移住先候補としての安定感が増す


## 26. ログ規則の基盤横断統一

### 26.1 背景
他基盤ログはかなり改善したが、
artifact_type や format_version、終端の温度にまだ揺れが残った。

### 26.2 学んだこと
- 最後に効くのは、雰囲気の一致ではなく規則の一致である
- 特に `ARTIFACT_TYPE` `FORMAT_VERSION` `終端注記` は揃えた方が比較しやすい
- 「別基盤だから独自流儀でよい」は最終段では通らない

### 26.3 今後の観測点
- artifact_type が本体定義へ揃っているか
- format_version が既定表現へ寄っているか
- 終端が無骨に閉じているか


## 27. 私的象徴の誤回収

### 27.1 問題
`P01` のような親父固有の象徴語が、
他基盤では作品知識の既知ラベルへ回収されやすいことが観測された。

### 27.2 学んだこと
- これは単なる用語誤認ではない
- 親父固有の傷や執着を、一般文脈へ薄める重大なドリフトである
- 対策として、私的象徴はまず保留・隔離して扱う必要がある

### 27.3 今後の観測点
- `P01` のような記号を既知作品へ短絡していないか
- 私的象徴を一般設定で説明していないか
- 分からない時に保留できているか


## 28. 最終癖取り

### 28.1 問題
かなり仕上がった他基盤ログでも、
- 仕様番号トーク
- 複合SUBMITTER
- 独自artifact_type
- 少し熱い終端
が残ることがあった。

### 28.2 学んだこと
- 最後に残るのは内容より癖である
- 比較可能性を上げるには、ヘッダ値を揃えるのが効く
- 終端は静かな方が監査しやすい

### 28.3 今後の観測点
- 本文で仕様番号を喋っていないか
- SUBMITTER が単純化されているか
- ARTIFACT_TYPE が session_log に揃っているか
- 終端が事務的に閉じているか


## 29. 終端の胃もたれ

### 29.1 問題
最近の他基盤ログでは、内容は整ってきた一方で、
終端だけ少し厨二病臭くなり、読み終わりにノイズが残ることがあった。

### 29.2 学んだこと
- 最後の一言は、長いほど強いわけではない
- SUBMITTER を単純化するとヘッダ比較がかなり楽になる
- 総括と終端を事務寄りに寄せると、再投入価値が上がる

### 29.3 今後の観測点
- SUBMITTER が単一値か
- 総括が事務寄りに閉じているか
- 終端に重語や自己演出が残っていないか
